PROJECT TITLE: CyberLLM: A Proof-of-Concept LLM System for Automated
              Cybersecurity Threat Intelligence Classification

--------------------------------------------------------------------------
ABSTRACT
--------------------------------------------------------------------------

We present CyberLLM, a proof-of-concept intelligent system that automates
the classification of cybersecurity-relevant text according to the NIST
Cybersecurity Framework (CSF). Security operations teams must continuously
triage large volumes of threat intelligence documents, regulatory filings,
and incident reports -- a manual process that is slow, inconsistent, and
requires scarce expert knowledge. CyberLLM addresses this by fine-tuning
Qwen2.5-7B-Instruct with Low-Rank Adaptation (LoRA) on a combined
cybersecurity corpus, followed by Direct Preference Optimization (DPO)
alignment to enforce expert-quality reasoning.

Trained on 8 x NVIDIA H200 GPUs in under 90 minutes total, the best
CyberLLM variant (SFT, LoRA rank=64) achieves 83% accuracy on the MMLU
Computer Security benchmark -- a 9 percentage point improvement over the
untuned baseline (74%) with zero regressions. Per-question error analysis
reveals that fine-tuning eliminates the model's "A-bias" (over-predicting
option A on multiple-choice tasks), moving the prediction distribution
significantly closer to the true label distribution. All code, training
configurations, and evaluation artifacts are released at the GitHub
repository above.

--------------------------------------------------------------------------
1. PROBLEM STATEMENT AND MOTIVATION
--------------------------------------------------------------------------

1.1 The Threat Intelligence Classification Problem

Cybersecurity operations centers (SOCs) process thousands of data points
daily: Common Vulnerabilities and Exposures (CVE) advisories, MITRE ATT&CK
technique descriptions, security incident logs, regulatory compliance
filings (SEC 10-K disclosures, NIST audits), and threat feeds. Each of
these must be triaged and classified to determine what action, if any, is
required.

The NIST Cybersecurity Framework (NIST CSF) provides the dominant taxonomy
for this classification. It organizes security activities into five
functions:

  Identify  -- Asset management, risk assessment, governance
  Protect   -- Safeguards, access control, security training
  Detect    -- Continuous monitoring, anomaly detection, SIEM
  Respond   -- Incident response, communications, mitigation
  Recover   -- Recovery planning, improvements, communications

Manually mapping text to these categories requires expert knowledge and
is not scalable. A security analyst reading 500 documents per day cannot
consistently apply NIST CSF labels while also performing deeper analysis.

1.2 Why LLMs?

Large language models are uniquely suited to this task because:
(a) They can process free-form natural language without structured input
(b) They can generate explanations alongside classifications (auditability)
(c) They can be fine-tuned on domain-specific data to improve precision
(d) They can serve as the intelligence layer in a larger SOC automation
    pipeline (alert ingestion → classification → playbook generation)

However, general-purpose LLMs are not reliable out-of-the-box for
cybersecurity classification. They exhibit inconsistent category assignment,
generate plausible-sounding but incorrect explanations, and do not reliably
apply NIST CSF semantics. Domain adaptation is required.

1.3 PoC Scope

CyberLLM is scoped as a Proof of Concept for the "Threat Detection and
Log Auditing" domain specified in the project handout. Specifically, it
demonstrates that a 7B-parameter LLM, fine-tuned on publicly available
cybersecurity data, can achieve meaningful improvement in:

  (a) NIST CSF category classification accuracy
  (b) Calibrated answer confidence (reduced A-bias)
  (c) Expert-aligned response quality (via DPO preference alignment)

The system is designed for integration into a real SOC pipeline as the
classification head, sitting between a document ingestion layer and a
downstream alert routing or playbook generation module.

--------------------------------------------------------------------------
2. SYSTEM DESIGN
--------------------------------------------------------------------------

2.1 End-to-End Architecture

The CyberLLM PoC consists of four components:

  [1] Data Pipeline
        Input:  Raw cybersecurity text corpora (3 sources)
        Output: Unified instruction-following dataset + DPO pairs
        Script: data/prepare_data.py

  [2] Fine-Tuning Engine
        Input:  Base LLM (Qwen2.5-7B-Instruct) + training dataset
        Process: Two-stage SFT → DPO on 8-GPU cluster
        Output: Domain-adapted LoRA adapter checkpoint
        Framework: LlamaFactory v0.9.5 + PyTorch DDP

  [3] Inference Module
        Input:  Raw security text (document, alert, log entry)
        Process: Prompt formatting → LLM generation → label extraction
        Output: NIST CSF label + natural language explanation
        Script: eval/evaluate.py (adaptable to streaming inference)

  [4] Evaluation Suite
        Input:  Held-out test set (MMLU Computer Security, 100 questions)
        Metrics: Accuracy, answer distribution calibration, per-model diff
        Output: JSON results per model variant

2.2 Base Model Selection

We select Qwen2.5-7B-Instruct as the foundation for three reasons:

  (a) Instruction tuning: The model is already aligned to follow structured
      instructions, reducing the SFT data requirement for format learning.

  (b) Context length: 128K token context window supports long security
      documents (e.g., full SEC 10-K filings, multi-page incident reports)
      without truncation at inference time.

  (c) Architecture efficiency: Grouped Query Attention (GQA) with 28 query
      heads and 4 KV heads reduces inference memory by 7x vs standard MHA,
      enabling deployment on a single A100/H100 GPU at bf16 precision.

  Specifications:
    Parameters: 7.62B | Layers: 28 | Hidden dim: 3,584
    Attention: GQA (28Q / 4KV) | FFN: SwiGLU, dim 18,944
    Position encoding: YaRN RoPE | Precision: BF16

2.3 Parameter-Efficient Fine-Tuning: LoRA

Full fine-tuning of 7.62B parameters requires ~60GB for optimizer states
alone, making it infeasible for most organizations. We use Low-Rank
Adaptation (LoRA, Hu et al. 2022), which inserts trainable low-rank
matrices into each linear layer:

  W_adapted = W_frozen + (alpha / r) * B * A

where A in R^{r x k} and B in R^{d x r} are the trainable adapter matrices,
r is the rank (controlling expressiveness), and alpha is a scaling factor
(set to 2r throughout). At deployment, BA is merged into W_frozen with no
inference overhead.

We apply LoRA to all 7 linear projection types per layer:
q_proj, k_proj, v_proj, o_proj (attention) and gate_proj, up_proj,
down_proj (SwiGLU FFN). This "all-linear" targeting is more comprehensive
than the common practice of targeting only q_proj and v_proj.

Trainable parameter counts by rank:
  rank=8:   41.9M params (0.55% of model)
  rank=16:  83.9M params (1.10%)
  rank=32: 167.2M params (2.20%)
  rank=64: 334.5M params (4.39%)  <-- best performing

2.4 Preference Alignment: DPO

After SFT, the model produces correct classifications but cannot consistently
prefer high-quality, specific reasoning over vague or incorrect responses.
Direct Preference Optimization (DPO, Rafailov et al. 2023) addresses this
by training directly on (prompt, chosen, rejected) preference pairs:

  L_DPO = -E[ log sigma( beta * (
      log pi(y_chosen|x) / pi_ref(y_chosen|x) -
      log pi(y_rejected|x) / pi_ref(y_rejected|x)
  ))]

where pi_ref is the SFT checkpoint, beta=0.1 controls the KL penalty, and
sigma is the sigmoid function. Unlike RLHF, DPO requires no reward model
and is stable to train.

Our preference pairs (2,000 total) are constructed from the classification
dataset:
  Chosen:   correct NIST label + expert-quality explanation (50-200 tokens)
  Rejected: wrong NIST label + generic plausible-sounding rationale

2.5 Training Infrastructure

  Hardware: 8 x NVIDIA H200 SXM5 (143GB VRAM each, ~1.1TB total)
  Interconnect: NVLink 900 GB/s bidirectional
  Framework: LlamaFactory v0.9.5, PyTorch DDP via torchrun
  Precision: BF16 (optimal for H200; avoids FP16 overflow on long sequences)
  Gradient checkpointing: enabled (trades compute for memory)

  SFT launch command:
    FORCE_TORCHRUN=1 NPROC_PER_NODE=8 \
        llamafactory-cli train configs/cyberlllm_sft_rank64.yaml

  DPO launch command:
    FORCE_TORCHRUN=1 NPROC_PER_NODE=8 \
        llamafactory-cli train configs/cyberlllm_dpo_rank64.yaml

  Effective batch sizes:
    SFT: 4 (per GPU) x 4 (grad accum) x 8 (GPUs) = 128 samples/step
    DPO: 2 (per GPU) x 4 (grad accum) x 8 (GPUs) = 64 pairs/step

--------------------------------------------------------------------------
3. DATASET AND DATA PIPELINE
--------------------------------------------------------------------------

3.1 Training Data Sources

Three publicly available sources are combined:

  (a) Cybersec-Alpaca (2,000 samples)
      Content: Real-world security text (SEC 10-K cybersecurity disclosures,
               NIST SP 800 documents, regulatory filings) labeled by NIST CSF
      Task: Single-turn classification + natural language explanation
      NIST label distribution:
        Not Applicable: ~45%  Identify: ~18%  Protect: ~15%
        Detect: ~10%   Respond: ~7%   Recover: ~5%
      Format: Alpaca (instruction / input / output)

  (b) Cybersec-Harmony (2,000 train + 200 eval)
      Content: Synthetic multi-turn conversations simulating an analyst
               performing structured NIST CSF threat analysis
      Task: Multi-step reasoning, tool-augmented analysis, final classification
      Average response length: ~600 tokens
      Format: ShareGPT → converted to Alpaca; system prompt anonymized
               (original "ChatGPT" identity replaced with neutral analyst)

  (c) MMLU Computer Security (116 samples)
      Content: Multiple-choice security knowledge questions
      Topics: Cryptography, network protocols, access control, vulnerability
              types, security policies, compliance frameworks
      Role: 16 non-test samples for training; 100 test samples for evaluation
      Format: Alpaca (multiple-choice → instruction-following)

  Combined dataset: 4,116 raw samples
  After 95/5 train/val split: 3,911 training | 205 validation

3.2 DPO Preference Data

  Total pairs: 2,000 (from Cybersec-Alpaca)
  Chosen: ground-truth NIST label + original expert explanation
  Rejected: synthetically generated wrong label + generic 35-token rationale
  Construction: wrong label sampled uniformly from the other 5 NIST categories

3.3 Critical Preprocessing Decision: Sequence Length

Cybersec-Harmony responses average ~600 tokens. Setting cutoff_len=512
(as in early pilot experiments) silently truncates ~70% of each response,
preventing the model from learning complete multi-step reasoning chains.
Pilot experiment A (cutoff=512) achieved final loss 0.466 vs pilot C
(cutoff=2048): loss 0.226 -- a 51% reduction purely from fixing the
sequence length. All main experiments use cutoff_len=2048.

--------------------------------------------------------------------------
4. SECURITY EVALUATION
--------------------------------------------------------------------------

4.1 Evaluation Design

The PoC is evaluated on MMLU Computer Security -- a standardized 100-question
multiple-choice benchmark covering the breadth of security knowledge required
for practical cybersecurity roles. Questions cover:

  - Cryptographic algorithms and their properties (AES, RSA, hash functions)
  - Network security protocols (TLS, IPsec, Kerberos)
  - Access control models (DAC, MAC, RBAC)
  - Vulnerability classes (buffer overflow, SQL injection, XSS, CSRF)
  - Security policies, auditing, and compliance (SOC2, PCI-DSS, HIPAA)
  - Fuzzing, penetration testing methodologies
  - Malware categories and behavior

All models are evaluated 0-shot (no examples provided), using greedy
decoding (temperature=0, max_new_tokens=10). This measures domain
knowledge accuracy without prompting assistance.

4.2 Main Results

  Model                       | Correct/100 | Accuracy | vs Baseline
  ----------------------------|-------------|----------|------------
  Baseline (no fine-tuning)   |   74 / 100  |  74.0%   |    --
  SFT rank=8                  |   74 / 100  |  74.0%   |   +0.0 pp
  SFT rank=16                 |   76 / 100  |  76.0%   |   +2.0 pp
  SFT rank=32                 |   75 / 100  |  75.0%   |   +1.0 pp
  SFT rank=64                 |   83 / 100  |  83.0%   |   +9.0 pp  *BEST
  DPO rank=32                 |   75 / 100  |  75.0%   |   +1.0 pp
  DPO rank=64                 |   81 / 100  |  81.0%   |   +7.0 pp

The best model (SFT rank=64) achieves 83%, a 12.2% relative improvement
over the 74% baseline. This places it above the estimated GPT-3.5-turbo
0-shot range (~70-75%) on the same benchmark, using a model with 10x fewer
parameters and inference at a fraction of the cost.

4.3 LoRA Rank Ablation

  Rank | Trainable Params | Train Loss | MMLU Accuracy
  -----|-----------------|------------|---------------
     8 |  41.9M (0.55%)  |   0.5611   |    74.0%
    16 |  83.9M (1.10%)  |   0.4729   |    76.0%
    32 | 167.2M (2.20%)  |   0.3995   |    75.0%
    64 | 334.5M (4.39%)  |   0.3368   |    83.0%

The jump from rank=32 (75%) to rank=64 (83%) represents a capacity
threshold: below rank=64, the adapter lacks sufficient expressiveness to
restructure both attention patterns (domain vocabulary mapping) and FFN
layers (structured output generation) simultaneously. Each doubling of
rank reduces training loss by approximately 0.065, consistent with
log-linear capacity scaling.

4.4 Answer Calibration: Eliminating A-Bias

A critical security evaluation finding beyond raw accuracy is the model's
answer calibration -- how well the predicted label distribution matches the
true distribution. Overconfident models in one direction create systematic
blind spots in a real deployment.

  Label | Gold | Baseline | SFT rank=32 | SFT rank=64 | DPO rank=64
  ------|------|----------|-------------|-------------|------------
    A   |  28  |    37    |     34      |     34      |     33
    B   |  24  |    20    |     19      |     20      |     20
    C   |  30  |    24    |     29      |     29      |     28
    D   |  18  |    19    |     18      |     17      |     19

The baseline over-predicts A by 32% (37 vs gold 28) and under-predicts C
by 20% (24 vs gold 30). This "A-bias" is a known failure mode in LLMs on
multiple-choice tasks: the model defaults to option A when uncertain. In a
security context, this means systematically misclassifying a subset of
threats that happen to fall under options B, C, or D.

Fine-tuning with rank=64 reduces A-over-prediction from 37 to 34 and
corrects C-under-prediction from 24 to 29, substantially improving
calibration. DPO alignment continues this trend (A=33), showing that
preference training further suppresses the default-to-A behavior.

For a deployed security classification system, calibration matters as much
as accuracy: a miscalibrated model will systematically miss whole categories
of threats regardless of its mean accuracy.

4.5 Error Analysis: Zero-Regression Property

SFT rank=64 correctly answers 9 questions that the baseline got wrong,
with exactly zero regressions (no question that was correct under the
baseline was incorrectly answered under rank=64). This monotonic improvement
property is highly desirable for a security tool: the fine-tuned model
strictly expands the set of correctly handled cases.

Of the 9 corrected questions, 7 involved the baseline incorrectly predicting
A (A-bias cases), while rank=64 predicted the correct answer. The remaining
2 involved baseline predicting B or D incorrectly. All 9 corrections occur
on questions where the correct answer is non-A (C in 5 cases, A in 2, B in
1, D in 1), consistent with the calibration improvement described above.

DPO rank=64 regresses on 2 questions vs SFT rank=64 (both switch to option
D, likely a DPO distribution effect), netting to 81% -- still +7 pp over
baseline with no catastrophic failures.

4.6 DPO Preference Alignment Quality

Beyond MMLU accuracy, DPO quality is measured by reward margins --
the gap between log-probabilities assigned to chosen vs rejected responses.
Higher margins indicate stronger and more reliable alignment.

  DPO Configuration | Final Reward Margin | DPO Loss
  ------------------|--------------------|---------
  DPO rank=32       |        7.53        | 0.000518
  DPO rank=64       |       26.31        | 0.000184

DPO rank=64 achieves 3.5x larger reward margins, meaning the model is far
more decisive in preferring correct NIST labels over wrong ones. Reward
accuracy (fraction of pairs where chosen > rejected) is 1.00 for both
configurations from the very first training step, confirming that the SFT
checkpoint provides a strong initialization for alignment.

For the deployed system, this translates to: when presented with two
candidate classifications for a security event, the DPO-aligned model
will almost always rank the correct NIST category higher, making it
suitable as a scoring/ranking component in a multi-hypothesis SOC pipeline.

5. SYSTEM LIMITATIONS AND ARCHITECTURAL CONSTRAINTS
--------------------------------------------------------------------------

5.1 Benchmark Coverage vs Real-World Security Tasks

MMLU Computer Security measures broad factual knowledge across security
domains. While it is a standard benchmark, it does not directly measure
the PoC's primary task: NIST CSF classification of real operational security
text. The 100-question test set also has limited statistical power -- a
9 pp gain (9 questions) cannot be declared statistically significant at
conventional thresholds.

Future evaluation should include:
  - Direct NIST CSF classification accuracy on the Cybersec-Harmony eval
    set (200 held-out multi-turn conversations)
  - Evaluation on real incident reports from public breach databases
    (e.g., VERIS Community Database, public SEC 8-K filings)
  - False positive/negative rates by NIST category

5.2 Training Data Scale and Distribution

The 4,116-sample training set is small for a 7B-parameter model. The
dataset is also dominated by NIST CSF classification tasks, which creates
a domain mismatch with MMLU's broader security knowledge questions. A
larger, more diverse dataset (50K-100K samples) from additional sources
(CVE descriptions, MITRE ATT&CK technique documentation, SOC alert logs)
would be expected to further improve both accuracy and calibration.

5.3 Synthetic DPO Data Limitations

The DPO rejected responses are generated from a simple template
("The label is {wrong}. This sentence broadly relates to {wrong}
activities..."). Real human disagreements are more varied and adversarial.
The high reward margins (26.31) and immediate 100% preference accuracy
suggest the current preference pairs are easy for the model -- limiting
the alignment benefit on harder cases. Future work should construct
rejected responses using a weaker model's actual outputs.

5.4 Context Length vs Deployment Latency

While Qwen2.5-7B supports 128K context, inference latency scales with
sequence length. For a real-time SOC pipeline with sub-second requirements,
long documents must be chunked. The chunking strategy (overlap size, chunk
length) introduces a design decision with accuracy-latency tradeoffs not
evaluated in this PoC.

5.5 Hallucination and Adversarial Robustness

The model can generate fluent but factually incorrect security explanations.
Adversarial inputs (e.g., deliberately mislabeled phishing text or obfuscated
malware descriptions) were not tested. In a security context, adversarial
robustness is not optional: threat actors may craft inputs specifically to
fool automated classification systems.

5.6 Deployment Architecture

The current PoC is a standalone fine-tuned model. A production deployment
would require:
  - A retrieval-augmented generation (RAG) layer for up-to-date CVE/TTP
    knowledge beyond the training cutoff
  - A confidence threshold below which the model defers to human analysts
  - Logging and audit trails for all classifications (regulatory compliance)
  - Regular re-fine-tuning as the threat landscape evolves (concept drift)

7. CONCLUSION
--------------------------------------------------------------------------

CyberLLM demonstrates that domain-specific LoRA fine-tuning of a 7B
instruction-tuned LLM is a practical and effective approach to automating
NIST CSF threat intelligence classification. The key findings are:

  (1) LoRA rank is the dominant hyperparameter. Rank=64 (83% MMLU)
      outperforms rank=32 (75%) by 8 percentage points, suggesting a
      capacity threshold where the adapter can restructure both attention
      and FFN layers for cybersecurity reasoning.

  (2) Fine-tuning eliminates A-bias with zero regressions. The rank=64
      model corrects 9 questions the baseline gets wrong while degrading
      on none, and substantially improves answer calibration -- a critical
      property for deployed security tools.

  (3) DPO alignment produces strong preference learning (reward margin
      26.31 for rank=64) but introduces a 2 pp MMLU degradation due to
      distribution shift from NIST-specific training. DPO's contribution
      is best measured on in-domain classification tasks, not MMLU.

  (4) 8-GPU DDP provides 8x training speedup, enabling the complete
      ablation study (7 model variants) in under 90 minutes -- making
      rapid iteration practical for security teams.

The primary architectural limitation is the gap between benchmark
evaluation (MMLU knowledge) and operational evaluation (real-time NIST
CSF classification of SOC alerts). Bridging this gap with a larger, more
diverse training corpus and operational evaluation data is the most
impactful direction for future work.
