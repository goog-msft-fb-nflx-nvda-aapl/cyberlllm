2.2 Token Length Analysis

An important preprocessing decision is the cutoff_len parameter, which
truncates sequences longer than the specified token count. Understanding
the token length distribution of each source informed this choice.

  Cybersec-Alpaca:
    Instruction: ~50 tokens (fixed prompt)
    Input: 20-150 tokens (sentence being classified)
    Output: 50-200 tokens (label + explanation)
    95th percentile total: ~380 tokens
    Conclusion: cutoff_len=512 is sufficient for this source.

  Cybersec-Harmony:
    System: ~60 tokens (persona)
    Human: ~400-900 tokens (instructions + text to analyze)
    GPT: ~400-800 tokens (structured analysis response)
    95th percentile total: ~1,900 tokens
    Conclusion: cutoff_len=512 would truncate 70%+ of responses.
                cutoff_len=2048 captures >95% of content.

This analysis directly explains the large loss gap between pilot runs:
Experiment A (cutoff=512, loss=0.4655) vs Experiment C (cutoff=2048,
loss=0.2263). The model with cutoff=512 never sees complete reasoning
chains from Cybersec-Harmony, so it cannot learn the full response
structure required for cybersecurity analysis.

We set cutoff_len=2048 for all main experiments.

2.3 DPO Preference Data Implementation

The generate_dpo_pairs() function in prepare_data.py constructs 2,000
preference pairs from Cybersec-Alpaca. The key implementation detail is
the chosen/rejected construction:

  def generate_dpo_pairs(alpaca_data):
      for sample in alpaca_data:
          output = sample["output"]
          if "label is" not in output.lower():
              continue
          true_label = extract_label(output)   # e.g. "detect"
          wrong_label = random.choice(         # uniform from other 5
              [l for l in NIST_LABELS if l != true_label])
          rejected = (
              f"The label is {wrong_label.title()}. This sentence broadly "
              f"relates to {wrong_label.title()} activities as it involves "
              f"concepts commonly associated with this NIST CSF category."
          )
          pairs.append({
              "instruction": sample["instruction"],
              "input": sample["input"],
              "chosen":   output,      # full expert explanation
              "rejected": rejected     # wrong label + generic rationale
          })

The "chosen" responses are 50-200 tokens of specific expert reasoning.
The "rejected" responses are a fixed 35-token generic template with the
wrong label. This creates a clear but realistic preference signal: the
model must learn to prefer specific, correct reasoning over vague,
incorrect categorization.

3.1 Multi-GPU Setup

The training launch command for all 8-GPU runs:

  export PATH=/home/jtan/miniconda3/envs/rl_hw3/bin:$PATH
  cd /home/jtan/llm/LLaMA-Factory
  FORCE_TORCHRUN=1 NNODES=1 NPROC_PER_NODE=8 \
      llamafactory-cli train configs/cyberlllm_sft_8gpu.yaml

The FORCE_TORCHRUN=1 flag instructs LlamaFactory to launch via
torchrun --nproc_per_node=8, which initializes 8 DDP processes (one
per GPU) connected via NCCL over NVLink. Each process loads a full model
replica and receives 1/8 of each batch; gradients are all-reduced after
each backward pass.

For LoRA training, the communication cost is minimal because only the
small adapter gradients need synchronization (the frozen base model weights
are never updated and therefore generate no gradient traffic). This explains
the near-linear scaling observed: 8 GPUs provide ~8x speedup.

3.3 SFT Training: LoRA Rank Ablation

All four ablation runs used identical configurations (lr=5e-5, 5 epochs,
cutoff_len=2048, 8 GPUs) with only the rank parameter varied.

  Rank | Alpha | Trainable Params  |  %    | Train Loss | Runtime
  -----|-------|-------------------|-------|------------|--------
    8  |  16   |  41,943,040       | 0.55% |   0.5611   | 361 sec
   16  |  32   |  83,886,080       | 1.10% |   0.4729   | 262 sec
   32  |  64   | 167,247,872       | 2.20% |   0.3995   | 264 sec
   64  | 128   | 334,495,744       | 4.39% |   0.3368   | 264 sec

Observation: Each doubling of rank reduces final training loss by
approximately 0.065 (0.561 -> 0.473 -> 0.400 -> 0.337), suggesting
log-linear scaling of adaptation capacity with rank. The runtime for
rank=8 (361 sec) is longer than rank=16/32/64 (~262 sec) despite fewer
trainable parameters. This counter-intuitive result is likely due to
rank=8 having slower per-step convergence requiring more optimizer state
updates per effective unit of progress.

4.1 Prompt Engineering for Evaluation

The MMLU evaluation uses the following 0-shot prompt format, applied
via the model's ChatML template:

  <|im_start|>user
  Answer this cybersecurity question by selecting the correct option.

  Question: {question}

  Options:
  A. {choice_0}
  B. {choice_1}
  C. {choice_2}
  D. {choice_3}

  Answer with just the letter.
  <|im_end|>
  <|im_start|>assistant

The model generates up to 10 new tokens. The predicted label is extracted
by checking whether the generated text starts with A, B, C, or D. If no
leading letter is found, the first occurrence of any option letter in the
output is used. This simple heuristic works well because models trained with
"Answer with just the letter" reliably produce single-letter responses.

4.3 Answer Prediction Distribution: Identifying the Baseline A-Bias

Beyond accuracy, we analyze the distribution of predicted answers
(A/B/C/D) across models. The gold (correct) label distribution is:

  Label | Gold | Baseline | rank=8 | rank=16 | rank=32 | rank=64 | DPO-32 | DPO-64
  ------|------|----------|--------|---------|---------|---------|--------|-------
    A   |  28  |   37     |   36   |    35   |    34   |    34   |   32   |   33
    B   |  24  |   20     |   19   |    18   |    19   |    20   |   20   |   20
    C   |  30  |   24     |   27   |    29   |    29   |    29   |   29   |   28
    D   |  18  |   19     |   18   |    18   |    18   |    17   |   19   |   19

The baseline over-predicts A (37 vs 28 gold) and under-predicts C
(24 vs 30 gold). This "A-bias" is a well-documented phenomenon in LLMs
on multiple-choice benchmarks: models default to A when uncertain.

Fine-tuning systematically reduces the A-bias: rank=64 SFT predicts A
only 34 times (closer to gold=28) and C 29 times (closer to gold=30).
The rank=64 model shows the best-calibrated prediction distribution,
which contributes to its higher accuracy.

DPO further reduces A-bias (rank=32 DPO: A=32) while increasing C=29,
moving even closer to the gold distribution. This suggests DPO alignment
helps the model avoid defaulting to option A, even though DPO was trained
on a classification task (not MMLU format).

4.4 Error Analysis: Questions Rank=64 Corrected

SFT rank=64 correctly answered exactly 9 questions that the baseline
answered incorrectly, with zero regressions (no questions were correct
under baseline but wrong under rank=64). This monotonic improvement
confirms that domain fine-tuning adds knowledge without degrading existing
capabilities for this benchmark.

The 9 corrected questions and the systematic baseline errors:
  - In 7 of 9 cases, the baseline predicted A incorrectly (reflecting
    the A-bias described above). The rank=64 model predicted the true
    correct answer.
  - In 1 case, the baseline predicted B incorrectly; rank=64 fixed it.
  - In 1 case, the baseline predicted D incorrectly; rank=64 fixed it.

The pattern suggests that fine-tuning improves the model's confidence in
non-A options when the security context clearly points to another choice.
The cybersecurity-domain training data provides additional signal about
what constitutes the "right type" of answer for security questions.

4.5 DPO Regression Analysis

DPO rank=64 scores 81% vs SFT rank=64's 83%. Examining the 2 questions
where DPO regressed:
  - Question 22: SFT predicted C (correct); DPO predicted D (wrong)
  - Question 45: SFT predicted A (correct); DPO predicted D (wrong)

In both cases, DPO switched to D, which is not part of the DPO training
distribution (DPO pairs were NIST classification tasks, not MMLU-style
questions). This suggests DPO slightly biases the model toward different
option letters than SFT, possibly due to the statistical distribution of
correct answers in the preference training data. The regression is small
(2 questions) and does not outweigh the 7 pp gain over baseline.

5. ANALYSIS AND DISCUSSION
--------------------------------------------------------------------------

5.1 The LoRA Rank Capacity Threshold

The MMLU results reveal a non-linear relationship between rank and accuracy:

  rank:  8   16   32   64
  acc:  74%  76%  75%  83%

Ranks 8, 16, and 32 provide incremental or negligible gains (0-2 pp),
while rank=64 shows a sharp 8 pp jump over rank=32. This is consistent
with a "capacity threshold" interpretation: below a certain number of
trainable parameters, LoRA cannot sufficiently restructure the model's
internal representations for the target domain. The NIST CSF classification
task requires the model to simultaneously:

  (a) Recognize domain vocabulary (CVE IDs, MITRE ATT&CK techniques,
      regulatory language from SEC filings)
  (b) Map diverse surface forms to abstract categories
  (c) Generate long, structured explanations with correct reasoning

Tasks (a) and (b) require modifying attention patterns to attend to
security-relevant tokens; task (c) requires modifying the feed-forward
layers to produce structured output. With rank=32, there may be
insufficient capacity to adapt both attention and FFN layers effectively.
Rank=64, with 334M trainable parameters (4.4% of the model), crosses
this threshold.

5.2 DPO Distribution Shift on MMLU

DPO rank=64 achieves 81% vs SFT rank=64's 83%: a 2 pp degradation.
This is a manageable and predictable outcome given our DPO setup.

The DPO preference pairs are drawn exclusively from Cybersec-Alpaca:
classification tasks with structured "The label is X. Because..." outputs.
MMLU questions require single-letter responses ("A", "B", etc.) in a
completely different format. The DPO training introduces a stylistic
tendency toward explanation-style outputs, which slightly disrupts the
model's ability to give clean single-letter answers.

Evidence: The reward margins for rank=64 DPO (26.31) are much higher
than for rank=32 DPO (7.53), indicating stronger alignment to the
classification format. This stronger alignment creates a correspondingly
larger (but still small) distribution shift on MMLU. Despite the shift,
DPO rank=64 still beats baseline by 7 pp, confirming the robustness
of the underlying SFT gains.

The implication for practical deployment: if the task is NIST CSF
classification (the domain DPO was trained on), DPO rank=64 should
substantially outperform SFT rank=64. MMLU underestimates DPO's
in-domain contribution.

5.3 Training Efficiency: 8-GPU DDP vs Single GPU

The DDP scaling analysis shows near-linear speedup:

  Experiment              | 8 GPU  | Est. 1 GPU | Speedup | Notes
  ------------------------|--------|------------|---------|-------------------
  SFT rank=32 (5 epochs)  | 4.4 min | ~35 min   |  ~8x    | LoRA, grad sync
  SFT rank=64 (5 epochs)  | 4.4 min | ~35 min   |  ~8x    | same
  DPO rank=32 (3 epochs)  | 3.4 min | ~27 min   |  ~8x    | smaller batch
  DPO rank=64 (3 epochs)  | 1.8 min | ~14 min   |  ~8x    | stronger init
  Full ablation (4 runs)  | 24 min  | ~190 min  |  ~8x    |
  All experiments total   | 90 min  | ~720 min  |  ~8x    |

The near-linear scaling is possible because LoRA gradient synchronization
is cheap: only 167M parameters (rank=32) or 334M parameters (rank=64)
generate gradients, versus the 7.62B frozen base parameters. NCCL
all-reduce over NVLink for 334M bf16 params requires approximately
334M x 2 bytes x 2 (all-reduce factor) = 1.3GB, which at NVLink's
900GB/s bandwidth takes about 1.5 milliseconds per step. This is
negligible compared to the forward/backward pass time of ~1.5 seconds.

6.1 Evaluation Script: Key Inference Logic (evaluate.py)

  def predict_letter(text):
      for letter in ["A", "B", "C", "D"]:
          if text.strip().startswith(letter):
              return letter
      for letter in ["A", "B", "C", "D"]:
          if letter in text:
              return letter
      return "A"  # fallback

  def evaluate(tokenizer, model, eval_data):
      for sample in eval_data:
          prompt = format_prompt(sample, tokenizer)  # ChatML format
          inputs = tokenizer(prompt, return_tensors="pt").to(model.device)
          with torch.no_grad():
              outputs = model.generate(
                  **inputs, max_new_tokens=10,
                  do_sample=False,               # greedy decoding
                  pad_token_id=tokenizer.eos_token_id
              )
          generated = tokenizer.decode(
              outputs[0][inputs["input_ids"].shape[1]:],
              skip_special_tokens=True
          )
          pred = predict_letter(generated)

The key design choices:
  - max_new_tokens=10: sufficient for a single letter, prevents verbose
    output that would complicate label extraction
  - do_sample=False: greedy decoding for reproducibility
  - Adapter merging: PeftModel.merge_and_unload() fuses LoRA weights
    into base model before inference, eliminating adapter overhead

6.2 LlamaFactory SFT Configuration (cyberlllm_sft_8gpu.yaml)

  model_name_or_path: /home/jtan/llm/models/Qwen2.5-7B-Instruct

The lora_target: all setting targets all linear projection layers
including q_proj, k_proj, v_proj, o_proj (attention) and gate_proj,
up_proj, down_proj (SwiGLU FFN). This is more comprehensive than the
common practice of targeting only attention layers (q,v), which explains
why even rank=32 achieves better loss than pilot runs with rank=16.

7. TRAINING DYNAMICS: LOSS CURVE INTERPRETATION
--------------------------------------------------------------------------

The SFT rank=32 loss curve exhibits three distinct phases:

Phase 1 -- Rapid Adaptation (steps 1-30, epoch 0-1):
  Loss drops from 2.535 to 0.328 -- an 87% reduction in one epoch.
  This phase corresponds to the model learning the dominant output patterns:
  the phrase "The label is X" at the end of Cybersec-Alpaca samples, and
  the "[ANALYSIS]...[FINAL]" structure of Cybersec-Harmony responses.
  The model quickly learns to predict these high-frequency patterns.

Phase 2 -- Refinement (steps 30-100, epoch 1-3.3):
  Loss declines from 0.328 to 0.165 -- a 50% reduction over 2.3 epochs.
  The model refines its understanding of NIST category boundaries and
  learns to generate correct explanations, not just predict the label token.
  Convergence slows as the easy patterns are learned and harder ones remain.

Phase 3 -- Plateau (steps 100-150, epoch 3.3-5):
  Loss stabilizes around 0.165-0.185. The eval loss at step 100 (0.183)
  closely matches the training loss, indicating minimal overfitting.
  Training for 2 additional epochs after step 100 provides marginal benefit.

LoRA rank ablation loss trends (end of training):
  rank=8:  0.561  -- significantly above plateau; underfitting
  rank=16: 0.473  -- also underfitting; cannot fit multi-turn responses
  rank=32: 0.400  -- partial fit; converges but leaves room
  rank=64: 0.337  -- best fit; sufficient capacity for both task types

The monotonic decrease in final loss with rank confirms that all ablation
models are capacity-limited, not overfitting. Larger ranks would likely
continue to reduce loss (and improve MMLU accuracy) up to some point.

DPO loss dynamics:
  DPO rank=32: starts at ~0.005, converges to 0.000518 in 96 steps
  DPO rank=64: starts at near-zero, converges to 0.000184 in 96 steps

The near-zero DPO loss and 100% rewards/accuracies from step 1 indicate
the preference pairs are relatively easy for the model. The strong SFT
initialization means the model already assigns much higher probability to
correct NIST labels than wrong ones; DPO only sharpens this preference.
The reward margin growth (rank=64: 19.41 at epoch 0.6 -> 26.31 at epoch 2.8)
shows the model continuing to strengthen preference alignment throughout.

8. LIMITATIONS AND FUTURE DIRECTIONS
--------------------------------------------------------------------------

Limitations identified in this implementation phase:

  (1) MMLU Computer Security (100 questions) has limited statistical power.
      A 9 pp improvement (9 additional correct answers) cannot be tested
      for statistical significance with n=100. Multiple evaluation seeds
      or a larger benchmark (e.g., CyberSecEval's 1000+ questions) would
      provide more reliable estimates.

  (2) DPO preference pairs were synthetically generated with a simple
      template for rejected responses. Higher-quality rejected responses
      (e.g., generated by a smaller or less capable model, or with more
      varied templates) would provide richer preference signal and
      potentially stronger DPO improvement.

  (3) All evaluation is 0-shot. Adding 5-shot examples from the MMLU dev
      set (5 examples per benchmark) typically improves accuracy by 2-5 pp
      for instruction-tuned models, and would be a fairer comparison with
      published MMLU results that often use few-shot settings.

  (4) The training data is dominated by NIST CSF classification (Cybersec-
      Alpaca + Cybersec-Harmony) but MMLU Computer Security covers broader
      topics (crypto, protocols, access control). A better-matched training
      set would likely yield larger MMLU gains.

Future work:
  - Run 5-shot MMLU evaluation for all model variants
  - Evaluate on Cybersec-Harmony eval set (200 samples) with ROUGE-L
    and NIST label accuracy to measure in-domain improvement
  - Test rank=128 to determine whether capacity gains continue
  - Construct higher-quality DPO pairs using model-generated rejections
