1. PROJECT OVERVIEW AND MOTIVATION
--------------------------------------------------------------------------

The volume of cybersecurity-relevant text that organizations must process
daily is enormous: vulnerability advisories, regulatory filings, threat
intelligence feeds, incident reports, and compliance documents. Manually
classifying and reasoning over this information using the NIST Cybersecurity
Framework (CSF) -- the dominant risk management standard -- is time-consuming
and requires specialized expertise. Automating this process with LLMs could
substantially reduce analyst workload and improve response times.

This project develops CyberLLM, a domain-adapted large language model for
cybersecurity threat intelligence. The model is fine-tuned to classify
security-relevant text into the five NIST CSF functional categories
(Identify, Protect, Detect, Respond, Recover) and to generate step-by-step
natural language explanations of its classifications. We go beyond simple
supervised fine-tuning by applying Direct Preference Optimization (DPO) to
align the model's output preferences with expert-quality reasoning.

The core challenge is that general-purpose LLMs -- even large, capable ones
like Qwen2.5-7B-Instruct -- are not optimized for cybersecurity domain
reasoning. They may misclassify industry-specific terminology, apply NIST
categories inconsistently, or generate explanations that are plausible but
technically incorrect. Domain-specific fine-tuning and preference alignment
are our proposed solutions.

Research Questions:
  RQ1: Does SFT on cybersecurity-specific data meaningfully improve
       classification accuracy over the vanilla pre-trained baseline?
  RQ2: Does DPO alignment further improve over SFT-only models?
  RQ3: What is the optimal LoRA rank for cybersecurity domain adaptation?
       (Ablation: rank in {8, 16, 32, 64})

2. RELATED WORK
--------------------------------------------------------------------------

2.1 LLMs for Cybersecurity

The application of LLMs to cybersecurity has grown rapidly. Ferrag et al.
(2023) surveyed LLMs across security domains, finding strong potential for
threat intelligence extraction and vulnerability analysis, while highlighting
the lack of domain-adapted models. Pearce et al. (2023) showed that GPT-3
and Codex can generate zero-day vulnerability patches with modest accuracy
when appropriately prompted. Koide et al. (2023) demonstrated that ChatGPT
can detect phishing URLs and content with competitive accuracy to classical
ML, but noted significant inconsistency in borderline cases.

For structured threat classification specifically, Mitra et al. (2023)
proposed LOCALINTEL, which uses LLMs to extract and structure threat
intelligence in STIX format, showing that domain context greatly improves
extraction quality. However, these approaches rely on prompting rather than
fine-tuning, limiting their adaptation to specialized frameworks like NIST CSF.

Our work differs by fine-tuning a 7B model specifically for NIST CSF
classification and by applying DPO alignment to improve output quality.

2.2 Parameter-Efficient Fine-Tuning

Full fine-tuning of a 7B model requires updating all ~7.6 billion parameters
and storing full optimizer states (~60GB in Adam fp32), making it memory-
prohibitive without large GPU clusters. Parameter-Efficient Fine-Tuning
(PEFT) methods reduce this cost substantially.

Adapter methods (Houlsby et al., 2019) insert small bottleneck modules
between transformer layers. While effective, they add inference latency.
Prefix tuning (Li and Liang, 2021) prepends trainable tokens to the key-value
cache; it is memory-efficient but can underperform for complex tasks.

LoRA (Hu et al., 2022) is the most widely adopted approach. It reparameterizes
weight updates as low-rank matrix products, adding no inference overhead when
the adapter is merged. QLoRA (Dettmers et al., 2023) extends LoRA to
quantized (4-bit) base models, enabling fine-tuning of 65B+ models on a
single GPU. We use LoRA without quantization, as our H200 GPUs have
sufficient VRAM for bf16 precision.

2.3 Alignment with DPO

Reinforcement Learning from Human Feedback (RLHF, Ouyang et al., 2022)
aligns LLMs by training a reward model on human preference data and then
applying PPO. This pipeline is complex, requires a separate reward model,
and can suffer from reward hacking.

Direct Preference Optimization (DPO, Rafailov et al., 2023) simplifies
alignment by showing that the optimal RLHF policy can be expressed in closed
form, eliminating the need for a reward model. Given (prompt, chosen,
rejected) triplets, DPO directly optimizes the language model. DPO has been
applied to instruction following, code generation, and summarization, but
its application to cybersecurity domain alignment is understudied.

2.4 Cybersecurity Benchmarks

The MMLU benchmark (Hendrycks et al., 2020) includes a computer_security
subset of 100 multiple-choice questions covering cryptography, network
security, access control, and compliance. It is a standard evaluation for
security knowledge across LLMs. CyberSecEval (Meta, 2024) provides a broader
evaluation suite including insecure code detection and prompt injection
resistance. We use MMLU computer_security as our primary benchmark due to
its accessibility and widespread use in the literature.

3. TECHNICAL APPROACH
--------------------------------------------------------------------------

3.1 Base Model: Qwen2.5-7B-Instruct

We use Qwen2.5-7B-Instruct (Alibaba Cloud, 2024) as our foundation model.

  Parameters: 7.62 billion
  Architecture: 28 transformer decoder layers
  Attention: Grouped Query Attention (GQA) with 28 query heads, 4 KV heads
  Hidden dimension: 3,584
  Intermediate dimension: 18,944 (FFN)
  Context window: 128K tokens (via YaRN RoPE extension)
  Instruction format: ChatML (Qwen template)
  License: Apache 2.0

Qwen2.5 uses Grouped Query Attention (GQA), which reduces KV cache memory
during inference by sharing key-value heads across multiple query heads. The
ratio of 28 query heads to 4 KV heads (7:1 sharing) reduces KV cache by 7x
versus standard Multi-Head Attention, making batch inference more efficient.

The model uses SwiGLU activation in the feed-forward layers and rotary
position embeddings (RoPE) for position encoding, consistent with the
LLaMA-family architecture.

3.2 Fine-Tuning Method: LoRA

LoRA (Hu et al., 2022) adapts a pre-trained weight matrix W_0 in R^{d x k}
by adding a trainable low-rank decomposition:

  W = W_0 + (alpha / r) * B * A

where A in R^{r x k} is initialized from a Gaussian distribution,
B in R^{d x r} is initialized to zero (so the adaptation starts at zero),
r is the rank, and alpha is a scaling hyperparameter (typically set to 2r).

At inference time, the product BA can be merged into W_0, adding no latency.
During training, only A and B are updated; W_0 is frozen.

For Qwen2.5-7B with rank=32:
  Target modules: q_proj, k_proj, v_proj, o_proj, gate_proj, up_proj, down_proj
  Trainable parameters: 167,247,872  (2.20% of 7.62B total)
  Frozen parameters: 7,453,163,520  (97.80%)

The LoRA rank controls the expressiveness of the adaptation:
  rank=8:  41,943,040 trainable params  (0.55%)
  rank=16: 83,886,080 trainable params  (1.10%)
  rank=32: 167,247,872 trainable params (2.20%)
  rank=64: 334,495,744 trainable params (4.39%)

3.3 Direct Preference Optimization

DPO (Rafailov et al., 2023) derives from the RLHF objective. Given a
reference model pi_ref (the SFT checkpoint) and a training policy pi_theta,
the DPO loss on a (prompt x, chosen y_c, rejected y_r) triplet is:

  L_DPO = -E[ log sigma( beta * (
    log(pi_theta(y_c|x) / pi_ref(y_c|x)) -
    log(pi_theta(y_r|x) / pi_ref(y_r|x))
  ))]

where beta is a KL penalty coefficient (we use beta=0.1) and sigma is the
sigmoid function. This loss encourages the policy to assign higher relative
probability to chosen responses over rejected ones, with the KL term
preventing excessive deviation from the reference model.

We use the sigmoid variant of DPO, which is more numerically stable than
the original log-sigmoid formulation for small datasets.

3.4 Training Framework: LlamaFactory + PyTorch DDP

LlamaFactory v0.9.5 provides the training orchestration layer. It handles:
  - YAML-based configuration parsing
  - Automatic torchrun launcher for multi-GPU DDP
  - Dataset preprocessing and tokenization
  - Gradient checkpointing, mixed precision (bf16)
  - Training curves, checkpoint saving, and evaluation

Multi-GPU training uses PyTorch DistributedDataParallel (DDP):
  - Each GPU holds a complete replica of the model
  - Gradients are synchronized (all-reduced) after each backward pass
  - With LoRA, only adapter gradients are communicated (much cheaper)
  - Effective batch size scales linearly with GPU count

Hardware configuration:
  GPUs: 8 x NVIDIA H200 (143,771 MiB VRAM each = ~1.1TB total)
  CUDA: 12.4, Driver: 550.163.01
  NVLink: H200 GPUs support 900 GB/s bidirectional NVLink bandwidth
  DDP backend: NCCL (NVIDIA Collective Communications Library)

3.5 Two-Stage Training Pipeline

Stage 1 -- Supervised Fine-Tuning (SFT):
  Objective: minimize next-token prediction loss on (instruction, response) pairs
  Loss: standard cross-entropy on response tokens only (instruction tokens masked)
  Effective batch size: 4 (per-GPU) x 4 (grad accum) x 8 (GPUs) = 128
  Epochs: 5, learning rate: 5e-5 (cosine schedule, 10% warmup)
  Sequence length: 2,048 tokens (cutoff_len)

Stage 2 -- Direct Preference Optimization (DPO):
  Initialize from: SFT checkpoint (adapter weights)
  Objective: maximize reward margin between chosen and rejected responses
  Effective batch size: 2 (per-GPU) x 4 (grad accum) x 8 (GPUs) = 64
  Epochs: 3, learning rate: 1e-5 (cosine schedule, 10% warmup)
  DPO beta: 0.1, loss type: sigmoid

4.1 Source Datasets

(a) Cybersec-Alpaca (2,000 samples)

This dataset contains security-relevant sentences drawn from real-world
documents (SEC 10-K cybersecurity disclosures, NIST SP 800 series documents,
regulatory filings). Each sample presents a sentence and requires the model
to: (1) assign the appropriate NIST CSF category, and (2) provide a natural
language explanation of the classification.

Format: Alpaca (instruction / input / output)
Label distribution across the 6 NIST CSF categories:
  Not Applicable: the majority class (~45%), reflecting that many
                  general business sentences are not CSF-relevant
  Identify:       ~18% (risk assessment, asset management topics)
  Protect:        ~15% (safeguards, access control, training)
  Detect:         ~10% (monitoring, anomaly detection)
  Respond:        ~7%  (incident response, mitigation)
  Recover:        ~5%  (recovery planning, restoration)

Sample input:
  Instruction: "Read carefully the sentence below and assign a category
  from the NIST cyber risk framework. Explain the reasoning."
  Input: "2022 Top Three MITRE Techniques: Remote Services (T1021),
  Lateral Movement, using Valid Accounts to log into remote services."
  Output: "The label is Detect. This sentence describes specific attacker
  techniques (MITRE ATT&CK T1021) which organizations should monitor for.
  Detecting lateral movement via Remote Services falls under the Detect
  function: implementing detection mechanisms for known adversarial TTPs."

(b) Cybersec-Harmony (2,000 training + 200 evaluation samples)

This dataset contains multi-turn conversations simulating a cybersecurity
analyst using the NIST CSF framework for structured threat analysis. Unlike
Cybersec-Alpaca (single-turn classification), Cybersec-Harmony includes
tool-calling and intermediate reasoning steps that reflect real analyst
workflows.

Format: ShareGPT (system / human / gpt conversation turns)
Preprocessing: The original system prompt identifies the assistant as
"ChatGPT, a large language model trained by OpenAI." We replace this with
a neutral cybersecurity analyst persona: "You are an expert cybersecurity
analyst. Analyze the provided text using the NIST Cybersecurity Framework.
Reason step by step." This prevents the fine-tuned model from adopting
a conflicting identity.
Average response length: ~600 tokens (much longer than Cybersec-Alpaca)

Sample structure (abbreviated):
  System: [neutral cybersec analyst persona]
  Human:  [DEVELOPER] You are a cybersecurity analyst assistant.
          Analyze the provided text according to NIST CSF...
          [Text to analyze]
  GPT:    [ANALYSIS] Analyzing the text for cybersecurity relevance.
          Checking NIST CSF alignment. Output: The label is Detect.
          The sentence describes monitoring of adversarial Remote Service
          techniques. [FINAL] Classification: Detect. Confidence: high.

(c) MMLU Computer Security (116 total samples)

The Massive Multitask Language Understanding (MMLU) benchmark includes a
computer_security task with 100 test, 11 validation, and 5 dev questions.
Topics covered: cryptography fundamentals, network security protocols,
access control models, common vulnerability types (buffer overflow, SQLi,
XSS), security policies, and compliance frameworks.

Format: 4-option multiple choice, converted to instruction-following.
Role: 100 test samples held out exclusively for evaluation; the remaining
16 samples (val + dev) converted to alpaca format for training.

Sample:
  Question: "Which of the following styles of fuzzer is more likely to
  explore paths covering every line of code in the following program?"
  Options: A. Generational  B. Blackbox  C. Whitebox  D. Mutation-based
  Answer: C. Whitebox
  (Whitebox fuzzers use coverage feedback and program analysis to
   maximize code path coverage.)

4.3 DPO Preference Data Construction

For DPO training, we require (prompt, chosen, rejected) triplets.
We derive these from the Cybersec-Alpaca classification task:

  Chosen: The ground-truth NIST label with the original correct reasoning
          explanation (e.g., "The label is Detect. This sentence describes
          monitoring for adversarial TTPs...")

  Rejected: A synthetically constructed incorrect classification:
            "The label is {wrong_category}. This sentence broadly relates
            to {wrong_category} activities as it involves concepts commonly
            associated with this NIST CSF category."

The wrong category is sampled uniformly from the other 5 NIST labels
(excluding the correct one). The rejected template is intentionally generic
and unconvincing, creating a clear preference signal.

Filtering: Only samples with a clearly extractable ground-truth label
("The label is X") are included. This yields 2,000 DPO pairs.

The DPO training signal teaches the model:
(a) To prefer specific, well-reasoned classifications over vague responses
(b) To prefer correct NIST CSF labels over plausible-sounding wrong ones
(c) To distinguish genuine domain knowledge from superficial pattern-matching

5. EVALUATION PROTOCOL

5.1 Primary Benchmark: MMLU Computer Security

We evaluate all models on the MMLU computer_security test set (100 questions)
using 0-shot prompting. The prompt format is:

  [User]: Answer this cybersecurity question by selecting the correct option.

  Question: {question}

  Options:
  A. {option_A}
  B. {option_B}
  C. {option_C}
  D. {option_D}

  Answer with just the letter.

  [Model]: {generated text}

The predicted label is extracted by checking if the generated text starts
with A, B, C, or D. Accuracy = (correct predictions) / 100.

7. PRELIMINARY RESULTS FROM PILOT EXPERIMENTS
--------------------------------------------------------------------------

Before the main experiments, three single-GPU pilot runs were conducted
to validate the setup and inform hyperparameter choices for the full study.
All pilots used the Cybersec-Alpaca dataset (2,000 samples) with 3 epochs.

  Exp | Rank | LR     | cutoff | Train Loss | Runtime   | Samples/sec
  ----|------|--------|--------|------------|-----------|------------
  A   |  8   | 1e-4   |  512   |   0.4655   |  310 sec  |  19.3
  B   |  16  | 5e-5   |  512   |  (n/a)     |    --     |   --
  C   |  16  | 5e-5   | 2048   |   0.2263   |  958 sec  |   6.3

Observations from pilot experiments:

(1) Context length is the dominant factor for loss convergence:
    Experiment C (cutoff=2048) achieves 51% lower final loss than
    Experiment A (cutoff=512), despite using the same dataset and
    similar hyperparameters. The Cybersec-Harmony responses average
    ~600 tokens, so cutoff=512 truncates most responses, preventing
    the model from learning complete reasoning chains.

(2) Lower learning rate improves optimization:
    Comparing runs with similar settings, lr=5e-5 produces smoother
    convergence and lower final loss than lr=1e-4. This is consistent
    with the recommendation for LoRA training to use smaller LRs than
    full fine-tuning, as the adapter parameters are more sensitive.

(3) Single-GPU is a clear bottleneck:
    Experiment C took 958 seconds (16 minutes) for 3 epochs on 2000 samples.
    With 8 GPUs and DDP, the effective batch size increases 8x and gradient
    synchronization is highly efficient over NVLink. The expected speedup
    for LoRA fine-tuning on DDP is close to linear (8x).

(4) Training loss of 0.23 indicates strong learning:
    A training loss of 0.23 on a language modeling task corresponds to
    approximately e^0.23 = 1.26 perplexity improvement over the previous
    token. This indicates the model has significantly adapted to the
    cybersecurity domain vocabulary and reasoning patterns.

These observations motivate our main experiment choices:
  - Use cutoff_len=2048 to preserve full response context
  - Use lr=5e-5 as the primary learning rate
  - Scale to 8 GPUs for ~8x speedup (enabling full ablation study)
  - Use rank=32 as primary (substantially more capacity than pilots)
  - Train for 5 epochs (more than pilots, using the larger dataset)
